Read this first — current status

Today the engine draws walls, monsters, items, and UI as procedural programmer-art (shaded polygons + glyphs), not PNGs. The sprite pipeline (src/render/sprites.ts, an atlas loader) is the planned hook and isn't wired yet. This guide tells you exactly what to draw so art drops in cleanly when that loader lands — every number is computed from the engine's real projection constants, so art authored to these sizes renders 1:1 with no scaling mush.

A Size cheat-sheet

Everything is drawn into a fixed 640 × 400 backbuffer that is then integer-scaled (×2, ×3, …) with smoothing off. Design at 1×: one art pixel = one game pixel.

B Pixel-art rules — non-negotiable

Work at 1×

The backbuffer scales up as a whole, so a 16px icon is genuinely 16 pixels. Never pre-scale your art.

No anti-aliasing

imageSmoothingEnabled = false. Hard 1px edges and dithering — never soft gradients.

Stay in palette

The whole game shares one 16-colour palette (below) so nothing clashes.

Alpha where it counts

Monsters, items, and icons need transparent backgrounds. Walls are opaque solid fills.

One light source

Top-left is the convention. Keep it consistent across every sprite.

Export

PNG, RGBA or indexed+alpha. Aseprite, Piskel, or GIMP all work.

C The palette — sweetie-16

Load these 16 colours in your editor and paint within them. Depth fog uses the palette, not alpha: near walls lighter (gray/slate), far walls darker (ink/navy).

D Wall textures — the hard part

There is no raycasting and no live texture-warping. Like Eye of the Beholder, the view is a painter's stack of pre-drawn, already-in- perspective sprites — one per frustum slot. The camera sees a 4-row-deep trapezoid of cells; each has a flat front face and two receding side faces. Your job is to draw them already foreshortened to fit these rectangles:

horizon · y≈134

Front faces (square)

    Front faces to scale (×0.55). Off-centre cells reuse the same sprite, shifted sideways — one front sprite per row, not per column.

    Side faces are trapezoids

    Sides recede toward the vanishing point, so they're tall on the near edge and short on the far edge: 682→300, 300→100, 100→60, 60→44 (rows 0–3). Author only the right side — the loader mirrors it left. A full brick set is only ~7 sprites: 3–4 front tiles + 3–4 side trapezoids.

    Making them look right

    1. Courses must converge. On a side trapezoid, angle the brick courses toward screen-centre (y≈134) — shorter and closer together toward the far edge. Parallel courses read as a flat billboard, not a receding wall.
    2. Tile seamlessly at cell edges. A front tile's right edge must line up with the next cell's left edge, or corridors show seams. Use a running-bond that continues across the boundary.
    3. Match texel density across depth. A brick should be the same fraction of the wall near and far, so both read as the same stone (~5 courses on the 100px tile → ~3 on the 60px tile).
    4. Bake in the fog. Paint each row a step darker (gray → slate → ink → navy). The darkening is what sells distance.
    5. Crisp corners. A 1px darker seam where front meets side reads as a wall corner.

    These swap into drawFrontFace() / drawSideFace() in viewport.ts; the exact rectangles come from frontRect() / sideQuad() in viewGeometry.ts. Doors reuse the front-face slot.

    Simpler alternative

    Too much to hand-draw? Author one square 64×64 tile per wall style and add a small code change to scale it into each front rect and skew it into each side quad. You lose foreshortening fidelity but draw far fewer sprites — this path needs an engine tweak.

    E Monsters & billboards

    Monsters are billboards — flat sprites standing on the cell floor, scaled by distance. Transparent background, anchored at the feet (bottom-centre sits on the floor).

    Supply either one large sprite (~128 tall) downscaled by the engine (simplest, slightly soft) or three crisp scale tiers (~82 / 52 / 36 — what the design targets). Author front / side / back poses; the engine picks one from the monster's facing. Add an idle bob and, optionally, a 2-frame walk and attack. The red hurt-flash is done in code — no "hurt" frame needed. Drawn by drawMonster().

    F Items · portraits · projectiles · UI

    Item icons — 16×16

    Transparent. Used in hand slots, the inventory grid, and on the floor (auto-scaled to ~14/11/8). Keep the silhouette readable at 8px. itemIcon.ts.

    Portraits — 24×24

    Transparent, in a 26px card box. Design them to gray out gracefully; the engine dims downed characters. partyPanel.ts.

    Projectiles — 4–11px

    Tiny by row (11/8/6/4). Dagger, magic bolt, sling stone. Transparent. drawProjectile().

    UI chrome — 9-slice

    Pane frames use a ~3px stone border. A 24×24 frame with 3px slices replaces the procedural bevel in chrome.ts.

    G Animated sprites — yes

    Fully supported. Rendering runs every frame (~60fps) independently of the 100ms simulation tick, so animation is smooth and timed in wall-clock milliseconds.

    Authoring

    • Lay frames out as a horizontal strip (frame 0, 1, 2… left to right), each frame the tier size — a 2-frame 82px walk is a 164 × 82 strip.
    • Keep counts small: 2–4 frames. It keeps the atlas tiny and reads authentically retro.
    • Timing: idle/walk ~150–250 ms/frame; attack a snappy 2-frame at ~80–120 ms.

    Advancing frames

    // pick a frame from wall-clock time, in the draw call
const frame = Math.floor(performance.now() / 200) % FRAME_COUNT; // 200ms/frame

// state-driven: branch on the AI state the monster already sets
const clip = monster.state === 'attack' ? attackClip : walkClip;

    What animates

    • Monsters — idle bob, walk, attack (frame sheets).
    • Doors — slide open/closed; already animated via door.progress, so a sprite door samples frames by progress.
    • Projectiles — hop cell-to-cell (hopMs); add a spin if you like.
    • Torches, water, pit shimmer — decorative 2–4 frame loopers.
    • Spell effects — short one-shot strips, played then discarded.

    Flash/tint overlays (hurt, heal, card damage-flash) are done in code — don't author frames for those.

    H Files & atlas

    Put source sheets in assets/ (walls/, monsters/, items/, ui/). Export each sheet as one PNG plus a JSON atlas mapping names to rectangles.

    {
  "brick_front_1": { "x": 0,   "y": 0, "w": 100, "h": 100 },
  "brick_side_1":  { "x": 100, "y": 0, "w": 100, "h": 100 },
  "skeleton_walk": { "x": 0, "y": 0, "w": 164, "h": 82, "frames": 2, "ms": 200 }
}

    Naming: <tileset>_<face>_<row> for walls (brick_front_1, crypt_side_2), <species>_<pose>_<anim> for monsters, <id> for items. The loader awaits the PNGs, slices per the JSON, mirrors right→left side walls, and hands named sub-images to the draw functions.

    I Workflow checklist

    1. Load the sweetie-16 palette; set canvas to the target size; turn off anti-aliasing.
    2. Walls: draw the front tiles (300/100/60/44), then the right side trapezoids.
    3. Verify courses converge and front tiles seam cleanly at cell edges.
    4. Darken each depth row a step for fog.
    5. Monsters/items: transparent backgrounds, tier sizes, anchored at the feet (monsters) or centred (items).
    6. Animations: horizontal strips, 2–4 frames; note ms/frame in the atlas.
    7. Export PNG + JSON into assets/; name per section H.
    8. Drop them in when the atlas loader lands — the draw functions are already sized to your art.
